Local MCP Gateway · 私有项目安全接入方案

把 ChatGPT 接到你的本地私有仓库，而不是把生产代码公开。

目标不是做一个炫技 App，而是做一个只给自己用的本地 MCP 网关：让我能安全读取项目、查看 diff、跑受控命令、生成证据报告；源码仍留在你的电脑和私有仓库里。

ChatGPT → 自定义 MCP Server → 本地私有项目 / Git / 测试命令 / 报告输出
为什么要做 MVP 工具 安全边界 阶段路线 使用场景

一、核心判断

生产代码不要为了让我能写 GitHub 而放到公有仓库。更合理的路线是：公有 GitHub 放 demo / HTML 报告 / 脱敏材料；真实生产项目通过本地 MCP 网关安全接入。
GitHub 模式

适合公开工作台

  • 公有 demo 仓库
  • HTML 展示页
  • Prompt / agent 配置模板
  • issue、PR、review、Actions 流程
本地 MCP 模式

适合私有生产项目

  • 读取本地项目结构和关键文件
  • 查看 git status / git diff
  • 运行白名单命令，例如测试和类型检查
  • 生成 patch、审计报告、阶段证据

二、一句话架构

ChatGPT
  ↓ 调用工具
你的 MCP Server
  ↓ 受控访问
本地私有仓库 / 脚本 / Git / 测试命令 / 文档生成

GitHub 工具是 OpenAI 已经帮你做好的远程开发工作台；自定义 MCP 是你自己定义“我可以调用哪些本地工具”。真正的价值是：代码不用公开，但我仍然可以基于受控证据做分析、复核和阶段裁决。

三、第一版 MVP：本地项目只读审计网关

第一版不要做全能执行器。先做只读和低风险证据工具，跑通后再逐步开放写入能力。

工具作用风险
list_projects列出允许访问的项目目录低
get_project_tree返回目录结构，过滤 node_modules、dist、.git低
read_file读取白名单项目内的指定文件中
search_files在项目中搜索关键词、函数名、错误信息中
git_status查看工作区状态低
git_diff查看当前变更 diff低
第一版不要开放 write_file、delete_file、git push、git commit、npm install、任意 shell 命令、读取整盘文件、读取 .env 或 SSH key。

四、建议项目结构

白名单配置示例

{
  "projects": [
    { "id": "wx-mst", "root": "D:/WX_MST" },
    { "id": "spplus-test", "root": "D:/spplus-agent-test" }
  ],
  "blockedPatterns": [
    ".env",
    ".env.*",
    "node_modules",
    ".git",
    "dist",
    "build",
    "*.pem",
    "*.key",
    "id_rsa",
    "secrets.*"
  ]
}

工具调用时只允许传 project_id 和项目内相对路径，不允许直接传 C:/Users/...、D:/... 这种任意绝对路径。

五、安全边界：不要做任意命令执行器

最危险的设计是 run_command(command: string)。这等于把你的电脑命令行暴露给模型，不能这么做。

正确方式是做白名单命令：

run_safe_command(project_id, command_id)

其中 command_id 只能是服务端预设值：

const commands = {
  typecheck: "npm run typecheck",
  test: "npm test",
  lint: "npm run lint",
  build: "npm run build"
};

路径防护

  • 只允许白名单项目
  • 禁止路径穿越
  • 禁止读敏感文件

命令防护

  • 不接受任意命令字符串
  • 只跑预设命令
  • 限制超时时间和输出大小

写入防护

  • 先只读
  • 后续只写报告目录
  • 源码改动必须人工确认

六、连接方式

本地启动 MCP Server。先在 Windows 本机跑起来，确认工具可以列项目、读文件、取 git diff。
用 Cloudflare Tunnel 暴露 HTTPS。第一版建议走 Tunnel，省掉证书和公网配置问题。
后续再换 IPv6 + Caddy。你已经有公网 IPv6 和 DDNS，稳定后可以用 Caddy 做 HTTPS 反代。
在 ChatGPT 里连接自定义 MCP。开启 Developer mode，创建 connector，填入 MCP endpoint。

七、阶段路线

阶段目标是否允许写源码
Phase 1本地 MCP 只读网关：项目列表、目录树、读文件、搜索、git status、git diff否
Phase 2安全命令执行：只跑 typecheck、test、lint、build 等白名单命令否
Phase 3生成 patch / evidence：输出补丁、测试日志、审计报告否
Phase 4受控写入 _phase_outputs、reports、generated不写业务源码
Phase 5源码 patch 应用，但必须人工确认，不能自动 push有限允许

八、跑通后的使用方式

你以后可以直接这样提需求：

检查 D:/WX_MST 当前原型有没有范围漂移。
请读取 scope 文档、查看页面结构、检查 git diff，最后输出 PASS / NEEDS_FIX / OUT_OF_SCOPE。

理想执行链路：

1. ChatGPT 调 list_projects 确认项目
2. 调 get_project_tree 看页面结构
3. 调 read_file 读取 scope / 关键页面
4. 调 git_status / git_diff 看变更
5. 调 run_safe_command 跑测试或类型检查
6. 输出裁决和下一步建议
这套模式的本质是：私有代码留在本地，ChatGPT 通过受控 MCP 工具拿证据，不靠猜。

九、最终收敛

最值得做的不是继续找更多 App，而是自建一个适合你的本地项目网关。

GitHub 是公开远程工作台；Codex/OpenCode 是本地代码执行器；自定义 MCP 是把 ChatGPT 安全接进你私有项目的桥。

ChatGPT + 本地 MCP + 私有仓库 + Codex/OpenCode

这条线一旦跑通，比“把生产代码放公有 GitHub 让我写”更安全，也更接近你想要的项目控制台形态。
